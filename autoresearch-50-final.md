# Autoresearch 50 — Ultimate Strategic Plan for Genius Team

## Executive Synthesis

This plan synthesizes the strongest ideas from:

- `competitive-analysis-opus.md`: correctly identifies friction, packaging, distribution, and trust gaps.
- `competitive-analysis-gpt5.4-challenge.md`: correctly rejects shallow mimicry, premature platform sprawl, and cargo-cult runtime work.
- `autoresearch-v21-plan.md`: provides useful growth ideas, but over-indexes on visibility before product sharpness.

### Strategic conclusion

Genius Team should not try to become "Superpowers but bigger."

Genius Team should become:

**the clearest, safest, fastest way to turn messy product intent into build-ready execution for both first-time builders and expert vibecoders.**

That means:

- Beginners need a guided path that removes jargon, reduces choices, and gets them to a visible result in minutes.
- Experts need a composable, low-friction, high-control system that prevents drift, preserves speed, and stays out of the way.
- The product must prove its core value before expanding distribution aggressively.
- The moat is not "48 skills." The moat is **adaptive orchestration + anti-drift governance + full-stack product execution**.

## Scoring Framework

The score below is the weighted strategic score across these 10 criteria:

1. Beginner experience
2. Expert experience
3. Installation experience
4. Time to first wow moment
5. Anti-drift & quality governance
6. Pipeline completeness
7. Skill composition & orchestration
8. Documentation & learning curve
9. Distribution & discoverability
10. Competitive differentiation

### Starting score

**5.2/10**

- Strong pipeline breadth and anti-drift concept.
- Weak onboarding, weak proof loops, weak packaging, weak discoverability.
- Too much surface area exposed too early.

## 9. `install.sh` Behavior

`install.sh` should do exactly this:

1. Detect OS and shell.
2. Verify required environment assumptions.
3. Install or update the Genius Team CLI.
4. Install the `Core` pack by default.
5. Ask the user to choose `Beginner`, `Builder`, or `Pro`.
6. Write mode config to a local Genius config file.
7. Run `genius doctor`.
8. Offer to launch the 5-minute challenge immediately.
9. If the user accepts, run `/genius-start` with the challenge prompt.
10. Print where logs, configs, and exported artifacts live.

### Installer rules

- It must support `--dry-run`.
- It must support non-interactive mode.
- It must be readable and auditable.
- It must fail with actionable messages.
- It must never silently overwrite user config.

## 10. Mode System

### Beginner mode

Purpose:

- Someone who has never coded before and wants a guided path to a working app.

Behavior:

- High narration
- Low jargon
- Always show next step
- Suggest templates automatically
- Block on critical missing artifacts
- Ask for approval at major milestones
- Generate visible dashboards by default

Default outputs:

- Plain-language explanations
- Project brief
- Draft spec
- Task checklist
- Launch checklist

### Builder mode

Purpose:

- Someone who has shipped side projects before and wants speed with structure.

Behavior:

- Medium narration
- Moderate autonomy
- Warnings instead of blocks for non-critical issues
- Optional dashboards
- Suggested workflows based on context

Default outputs:

- Brief summaries
- Build-ready specs
- Architecture plan
- Sequenced execution tasks

### Pro mode

Purpose:

- Someone shipping production software frequently who wants minimal overhead and strong governance.

Behavior:

- Low narration
- High autonomy
- Compact summaries
- Risk-first reporting
- Non-blocking warnings except on severe issues
- No dashboard generation unless requested
- Fast path commands and terse review lanes

Default outputs:

- Concise diffs
- Assumptions and blockers
- Validator summaries
- Review reports
- Exportable markdown artifacts

### Mode switching rules

- Users can switch modes at any time with `/genius-mode`.
- Existing sessions preserve artifacts but adapt approval, narration, and guard behavior.
- Validators keep the same underlying standards, but enforcement levels vary by mode.
